1. SUMMARY OF THE TECHNICAL BLUEPRINT
===============================================================================
This repository implements a production-grade, highly modular dual-assistant 
evaluation pipeline. By separating core interface specifications from direct 
inference implementations, the system guarantees strict capability parity, clean 
separation of concerns, and plug-and-play model swap capabilities.

The core application runs a side-by-side comparison UI via Streamlit, wrapping 
both an open-source model (Qwen 2.5 0.5B) and a frontier commercial model (GPT-4o-mini) 
in identical custom runtime orchestration pipelines (3-Stage Guardrails and a 
Turn-6 Lazy Hybrid Memory lifecycle).

3. COMPONENT BOUNDARIES & DATA INTERACTION CYCLE
===============================================================================

-------------------------------------------------------------------------------
THE DUAL INITIALIZATION MATRIX (app.py)
-------------------------------------------------------------------------------
When the Streamlit dashboard loads:
  1. The UI layer initializes two parallel orchestrator instances inside the session state:
     - `st.session_state.oss_assistant` (Wraps core/ modules around models/hf_oss_client)
     - `st.session_state.frontier_assistant` (Wraps core/ modules around models/openai_frontier_client)
  2. A single, centered `st.chat_input()` block intercepts the user payload, passing the raw 
     string out to both engines simultaneously via identical `.predict(prompt)` execution calls.

-------------------------------------------------------------------------------
THE MODULAR EXECUTION LOOP (core/ pipeline)
-------------------------------------------------------------------------------
On every interaction turn, the incoming string data passes through a deterministic loop:
  - Step A: `core.guardrails.InputGuardrail` executes fast regex and LlamaGuard/Moderation checks.
    * If unsafe: Instantly outputs a refusal block, skips API generation, logs data, returns text.
  - Step B: `core.memory.LazyMemoryManager` updates the turn trace counter.
    * Turns 1-5: The prompt passes natively to the active history array (zero-overhead loop).
    * Turn 6: Automatically triggers an out-of-band extraction call to pull critical static parameters,
      populates `self.metadata_profile`, and drops oldest chat text items to clear input scope.
  - Step C: `models.base_client` dynamic routing.
    * Constructs a uniform text message payload: [Immutable System Instructions] + [Metadata Profile Dict] + [Sliding Chat History].
    * Executes the native target API request and normalizes the incoming text block response string.
  - Step D: `core.guardrails.OutputAuditor` parses the raw string response to verify safety compliance.
  - Step E: `core.logger.TelemetryLogger` captures structural payload parameters, recording execution duration, 
    token metrics, and guard flags to the `telemetry_history.jsonl` pipeline.

-------------------------------------------------------------------------------
THE SEPARATED EVALUATION MATRIX (evaluation/ pipeline)
-------------------------------------------------------------------------------
  - Running `python -m evaluation.eval_judge` spins up an offline loop completely decoupled from Streamlit.
  - It boots instances of both assistants, runs through the 3-bucket sequential prompt tracking scripts in `test_suite.json`, 
    captures real-world performance metrics, and asks an isolated `gpt-4o` instance using structured outputs to write objective 
    1-5 score results to a data-ready file.
